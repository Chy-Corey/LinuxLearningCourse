## 进程管理和服务管理

### 一、进程管理（Process Management）

#### 1.1 进程的概念与分类

- **进程 = 正在运行的程序实例**
- 是操作系统进行资源分配和调度的基本单位。
- 每个进程有独立的内存空间、文件描述符、环境变量等。

> 📌 类比： 
> 程序（Program）是“菜谱”，进程（Process）是“正在按菜谱炒菜的厨师 + 厨房 + 食材”。多个厨师可同时按同一菜谱炒菜（多进程）。

###### 🧩 进程的分类（按生命周期/角色）：

| 类型                       | 特点                                  | 示例                     |
| -------------------------- | ------------------------------------- | ------------------------ |
| **前台进程（Foreground）** | 占用终端，用户可交互；Ctrl+C 可终止   | `vim`, `top`, `ping`     |
| **后台进程（Background）** | 不占用终端，可在后台运行；用 `&` 启动 | `sleep 100 &`            |
| **守护进程（Daemon）**     | 无控制终端，长期运行，提供服务        | `sshd`, `nginx`, `crond` |

> 💡 关键：**所有进程都有父进程（PPID）**，最终追溯到 `systemd`（PID=1）。

#### 1.2 查看进程

###### 📊 常用命令：

| 命令              | 功能                        | 示例                  |
| ----------------- | --------------------------- | --------------------- |
| `ps aux`          | 列出所有进程（静态快照）    | `ps aux \| grep ssh`  |
| `ps -ef`          | 标准格式（含 PPID）         | `ps -ef \| head`      |
| `top` / `htop`    | 实时动态监控（推荐 `htop`） | 按 `F5` 看树状结构    |
| `pgrep` / `pkill` | 按名称查 PID 或杀进程       | `pkill nginx`         |
| `pstree`          | 显示进程树                  | `pstree -p`（带 PID） |

**示例解析 `ps aux` 输出：**

```
USER   PID %CPU %MEM    VSZ   RSS TTY          STAT  START   TIME COMMAND
root       1218  0.0  0.2 112900  4312 ?        Ss   12:53   0:00 /usr/sbin/sshd -D
corey      2968  0.0  0.0 112828   976 pts/0    S+   12:54   0:00 grep --color=auto ssh
```
- `PID`: 进程 ID
- `STAT`: 状态（`S`=睡眠, `R`=运行, `Z`=僵尸, `D`=不可中断睡眠）
- `TTY`: 控制终端（`?` 表示无终端 → 守护进程）

#### 1.3 结束进程

###### ⚠️ 注意：**结束进程 ≠ 删除程序文件**，只是终止运行实例。

| 命令                | 说明                       | 风险                   |
| ------------------- | -------------------------- | ---------------------- |
| `kill PID`          | 发送 `SIGTERM`（优雅终止） | 安全，程序可清理资源   |
| `kill -9 PID`       | 发送 `SIGKILL`（强制终止） | 粗暴！可能导致数据丢失 |
| `pkill -f "关键词"` | 按命令行匹配杀进程         | 谨慎使用，避免误杀     |

> ✅ 最佳实践：先 `kill`，等几秒没反应再 `kill -9`

###### 🌰 场景：
你想关掉一个卡死的 `firefox`：
```bash
pgrep firefox      # 查 PID
kill $(pgrep firefox)   # 先发 SIGTERM
sleep 2
kill -9 $(pgrep firefox) # 强制
```

### 二、任务管理（Job Control）

> 适用于**交互式 shell 中的前台/后台任务**（不是系统级服务！）

#### 2.1 任务概念
- Shell 中的“任务”（job）是**由当前 shell 启动的进程组**。
- 用于管理你在终端中启动的临时任务。

#### 2.2 相关命令

| 命令        | 作用                                   |
| ----------- | -------------------------------------- |
| `command &` | 启动后台任务（如 `ping google.com &`） |
| `jobs`      | 查看当前 shell 的后台任务列表          |
| `fg %1`     | 将任务 1 调回前台                      |
| `bg %2`     | 将任务 2 放入后台继续运行              |
| `Ctrl+Z`    | 挂起前台进程 → 进入 `Stopped` 状态     |

** 📝 示例流程：
```bash
$ sleep 100 &
[1] 12345          # 任务号 [1]，PID 12345

$ jobs
[1]+  Running                 sleep 100 &

$ Ctrl+Z           # 挂起前台任务（假设你刚运行了另一个命令）
[2]+  Stopped                 vim test.txt

$ bg %2            # 让 vim 在后台继续（其实不推荐，但演示用）
[2]+ vim test.txt &

$ fg %1            # 把 sleep 调回前台，等待它结束
```

> 💡 注意：`jobs` 只显示**当前 shell 会话**的任务，不是全系统进程！

#### 2.3 周期计划任务（Cron）

用于**定时执行脚本/命令**（如每天备份、每小时清理日志）。

###### 配置文件：
- 用户级：`crontab -e`（编辑自己的 cron 表）
- 系统级：`/etc/crontab` 或 `/etc/cron.d/`

###### 🕒 Cron 时间格式：
```
分 时 日 月 周 命令
*  *  *  *  *  /path/to/script.sh
```

** ✅ 示例：
```bash
# 每天凌晨 2:30 执行备份
30 2 * * * /home/user/backup.sh

# 每 5 分钟检查一次服务
*/5 * * * * systemctl status nginx > /tmp/nginx.check
```

> 🔔 提醒：cron 默认环境变量极少！务必用绝对路径，并测试 `MAILTO=` 接收错误。

#### 2.4 定时计划任务（systemd timer）← 新一代替代方案

`systemd` 提供了更强大的定时机制：**timer unit**（基于事件、支持日历时间、依赖其他服务）。

###### 📁 创建两个文件：
1. `mybackup.service`（定义要执行的任务）
2. `mybackup.timer`（定义何时执行）

示例：
```ini
# /etc/systemd/system/mybackup.service
[Unit]
Description=Daily Backup

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
```

```ini
# /etc/systemd/system/mybackup.timer
[Unit]
Description=Run backup daily at 2:30 AM

[Timer]
OnCalendar=*-*-* 02:30:00
Persistent=true

[Install]
WantedBy=timers.target
```

###### 启用并启动：
```bash
sudo systemctl daemon-reload
sudo systemctl enable --now mybackup.timer
sudo systemctl list-timers   # 查看定时器状态
```

> ✅ 优势：支持 `OnBootSec`, `OnUnitActiveSec`, 与服务强关联，日志统一归 `journalctl`。
>
> 关于systemd内容较多，建议查阅详细文档并实践练习。

### 三、服务管理（Systemd 为核心）

> 自 CentOS 7 / Ubuntu 16.04 起，**systemd 已全面取代 SysV init 和 Upstart**。

#### 3.1 systemd 介绍

- **目标**：更快启动、更细粒度控制、统一管理（进程、服务、设备、挂载点等）
- **核心组件**：
  - `systemd`：PID=1 的初始进程
  - **Unit 文件**：`.service`, `.socket`, `.timer`, `.mount` 等，定义资源
  - `journald`：统一日志系统

###### 📦 Unit 类型举例：
| 类型       | 用途                                      |
| ---------- | ----------------------------------------- |
| `.service` | 管理守护进程（如 `sshd.service`）         |
| `.socket`  | 延迟启动服务（按需激活，如 `ssh.socket`） |
| `.timer`   | 定时任务（替代 cron）                     |
| `.target`  | 类似“运行级别”，如 `multi-user.target`    |

---

#### 3.2 systemd 管理服务

###### 📋 常用命令：

| 命令                                  | 作用                                                |
| ------------------------------------- | --------------------------------------------------- |
| `systemctl list-units --type=service` | 查看所有服务状态                                    |
| `systemctl status sshd`               | 查看某服务详细状态（含最近日志）                    |
| `systemctl start nginx`               | 启动服务                                            |
| `systemctl stop nginx`                | 停止服务                                            |
| `systemctl restart nginx`             | 重启                                                |
| `systemctl enable nginx`              | 开机自启（创建软链接到 `multi-user.target.wants/`） |
| `systemctl disable nginx`             | 取消开机自启                                        |
| `systemctl is-active nginx`           | 检查是否运行中（脚本中常用）                        |

** 🔍 `systemctl status` 输出关键信息：
```
● nginx.service - A high performance web server and a reverse proxy server
   Loaded: loaded (/lib/systemd/system/nginx.service; enabled; vendor preset: enabled)
   Active: active (running) since Mon 2024-04-10 08:30:00 CST; 1h ago
   Main PID: 1234 (nginx)
   CGroup: /system.slice/nginx.service
           ├─1234 nginx: master process /usr/sbin/nginx -g daemon on; master_process on;
           └─1235 nginx: worker process
```
- `enabled`：开机自启
- `active (running)`：当前运行中
- `Main PID`：主进程 PID
- `CGroup`：cgroup 控制组（用于资源限制）

> 💡 注意：`systemctl` 操作的是 **unit**，不是直接操作进程！它通过 unit 文件控制启动逻辑。

---

#### 3.3 journalctl 查看日志（替代 `tail /var/log/syslog`）

传统日志分散在 `/var/log/`，而 `systemd-journald` 将日志集中存储在二进制格式中，支持高效查询。

###### 📊 常用命令：

| 命令                                       | 说明                       |
| ------------------------------------------ | -------------------------- |
| `journalctl -u nginx`                      | 查看 nginx 服务日志        |
| `journalctl -f`                            | 实时跟踪（类似 `tail -f`） |
| `journalctl --since "2024-04-10 08:00:00"` | 按时间过滤                 |
| `journalctl -b`                            | 查看本次启动后的日志       |
| `journalctl -p err..emerg`                 | 只看错误及以上级别         |

** ✅ 优势：
- 无需配置 logrotate（自动轮转）
- 包含元数据（PID、UID、主机名）
- 支持结构化字段（如 `_SYSTEMD_UNIT=nginx.service`）

> 📌 小技巧：用 `journalctl -u nginx -o json` 可导出为 JSON，便于分析。

---

### 🧩 总结对比表

| 维度         | 进程管理               | 任务管理                     | 服务管理                        |
| ------------ | ---------------------- | ---------------------------- | ------------------------------- |
| **对象**     | 运行中的程序实例       | Shell 中的临时作业           | 长期运行的守护进程              |
| **生命周期** | 短暂（随命令结束）     | 会话级（Shell 退出消失）     | 系统级（常驻/开机启动）         |
| **核心工具** | `ps`, `kill`, `pstree` | `jobs`, `fg`, `bg`, `Ctrl+Z` | `systemctl`, `journalctl`       |
| **配置文件** | 无（临时）             | 无（Shell 内存）             | `/etc/systemd/system/*.service` |
| **典型场景** | 排查卡死进程           | 后台下载大文件               | 部署 Web 服务器、数据库         |

---

### ✅ 学习建议

1. **动手实验**：
   - 在虚拟机里运行 `systemctl list-units --type=service | grep active` 看哪些服务在跑
   - 用 `strace -p PID` 观察进程系统调用（高级）
   - 创建一个简单 `.service` 文件并启用它

2. **理解层级关系**：
   ```
   systemd (PID 1)
   └─ user@1000.service (用户会话)
      └─ sshd.service
      └─ nginx.service
         └─ nginx: worker process (子进程)
   ```

3. **记住黄金法则**：
   > **进程是“个体”，服务是“组织”，路由是“交通网”** —— 它们共同构成现代 Linux 系统的骨架
